# 网络舆情事件智能分析系统 — 课程设计报告

> **课程名称**：大型程序设计实践
> **项目题目**：网络舆情事件智能分析系统
> **团队成员**：5 人

---

## 一、课程设计目的

本课程设计旨在综合运用 Python 程序设计、数据结构与算法、Web 开发、机器学习、自然语言处理等课程知识，构建一个完整的**网络舆情事件智能分析系统**。该系统从微博、B站、知乎、今日头条等 11 个信息源自动采集舆情数据，经清洗、分词、事件聚合、情感分析后，对舆情事件进行生命周期预测、传播路径溯源、虚假文本检测和跨事件因果分析，最终以 Web 可视化界面呈现分析结果。

通过本项目的实践，达到以下目标：

1. **掌握大型软件系统的分层架构设计**：从数据采集层、数据处理层、自然语言处理层、算法分析层到 Web 服务层的完整六层技术栈；
2. **实践多种机器学习与统计方法在真实场景中的工程化应用**：包括 TF-IDF 特征工程、逻辑回归分类、BGE 向量嵌入、FAISS 语义检索、Holt 指数平滑预测、PageRank 图算法、格兰杰因果检验等；
3. **培养团队协作与工程化思维**：5 人团队按模块并行开发，通过统一 JSON Schema 实现跨模块数据对接，体验真实工业级协作流程；
4. **理解算法可解释性的工程实现**：SHAP 词级解释、反事实推演因果量化、临界减速预警等，使黑盒算法的输出对用户可理解、可追溯。

## 二、需求分析

### 2.1 项目背景

当前突发公共事件、社会热点话题在社交平台上传播迅速，舆情信息可能在短时间内对社会稳定产生重大影响。如何从海量多源信息中自动识别热点事件、分析传播趋势、发现虚假信息、预判舆情走向，是网络空间治理的现实需求。

### 2.2 功能需求

系统共需实现 **7 项基础功能 + 2 项高级功能 + 1 项创新功能**：

| 序号 | 功能模块 | 需求级别 | 说明 |
|:---:|------|:---:|------|
| 1 | 多平台网络爬虫 | 基础 | 从微博、B站、知乎、今日头条、新闻网站等 11 个平台采集舆情数据 |
| 2 | 数据清洗与预处理 | 基础 | 正文抽取、去重(BBone/MD5/SimHash)、繁简转换、中文分词、TF-IDF 特征提取 |
| 3 | 热点事件发现与聚合 | 基础 | 用 BGE 向量嵌入 + FAISS 聚类将同一事件的多条报道聚合；支持相似事件检索 |
| 4 | 情感倾向分析 | 基础 | 对事件文本进行正面/负面/中性情感计算，输出情感时间线 |
| 5 | 舆情事件分类 | 基础 | 自动对事件进行主题分类（社会民生/科技/财经/国际等） |
| 6 | 舆情生命周期预测 | 基础 | 判断事件所处阶段（潜伏/成长/高潮/衰退/二次爆发），预测未来 24 小时热度走势 |
| 7 | LLM 事件要素抽取 | 基础 | 用大语言模型提取事件摘要（时间、地点、主体、核心信息） |
| 8 | 虚假文本检测 | 高级 | 对采集信息进行真伪判定（可信/待验证/疑似虚假），输出 SHAP 词级解释 |
| 9 | 传播路径溯源 | 高级 | 识别传播关键节点，构建传播链路图，量化节点因果影响力 |
| 10 | 跨事件因果分析 | 创新 | 发现多个舆情事件之间的格兰杰因果和传递熵因果关系 |

### 2.3 界面需求

| 界面 | 功能 |
|------|------|
| 首页热点看板 | 展示全部舆情事件卡片列表，按热度排序；预警分布概览；事件因果有向图 |
| 事件详情页 | 阶段进度条、热度大数字、趋势预测折线图 + 置信区间带、四维雷达图、预警信号灯 |
| 传播分析模块 | 力导向传播拓扑图、关键节点评分卡片、反事实推演排行、分平台触达图 |
| 文章判假模块 | 文章列表 + 三色判定标签 + SHAP 词级高亮 + 特征贡献柱状图 |
| 智能问答 | 支持自然语言提问，回答阶段/趋势/风险/虚假信息等问题 |

### 2.4 非功能需求

- **性能**：算法模块纯 CPU 运行，不依赖 GPU；模型文件 < 100KB，首次加载 ≤ 30 秒；
- **可靠性**：数据质量自动四级分级（full/basic/minimal/sparse），低质量数据自动降级算法功能；
- **可扩展性**：Pipeline 统一接口设计，新增算法模块无需修改已有代码；
- **可解释性**：所有算法判定附带可追溯的解释依据。

### 2.5 开发环境

- **语言**：Python 3.10+
- **Web 框架**：FastAPI + Uvicorn
- **前端**：Vue 3 + ECharts + D3.js
- **关键依赖**：numpy, scipy, scikit-learn, networkx, ruptures, jieba, OpenCC, readability-lxml, sentence-transformers, faiss-cpu
- **版本管理**：Git + GitHub（三个分支并行开发）

```
│                  Web 服务层 (4号 server.py)                        │
│    FastAPI   │   /api/events   /api/event/{id}                   │
│              │   /api/articles/check   /api/cross-event           │
└──────────────────────────────┬───────────────────────────────────┘
                               │ Pipeline 统一接口
┌──────────────────────────────┴───────────────────────────────────┐
│              算法分析层 (4号 time_series/)                         │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐ ┌─────────────────┐ │
│  │lifecycle/│ │propagation│ │fake_detection│ │ cross_event.py  │ │
│  │生命周期   │ │/ 传播溯源 │ │/ 虚假检测    │ │ 跨事件因果      │ │
│  └──────────┘ └──────────┘ └──────────────┘ └─────────────────┘ │
└──────────────────────────────┬───────────────────────────────────┘
                               │ 标准化 JSON
┌──────────────────────────────┴───────────────────────────────────┐
│               自然语言处理层 (3号 dev 分支)                         │
│  BGE嵌入 │ 情感推理 │ 事件聚类 │ 事件分类 │ 热度计算 │ LLM提取    │

┌──────────────────────────────┴───────────────────────────────────┐

│  微博 │ B站 │ 知乎 │ 今日头条 │ 百度新闻 │ NewsAPI │ MediaCloud   │
```

**数据流说明**：

1. **1号爬虫** → 从 11 个平台采集原始数据（HTML/JSON），输出 `raw_data.json`
2. **2号预处理** → 7 步流水线处理，输出 `segmented_data.csv` + `advanced_analysis.csv`
3. **3号 NLP** → 18 个处理脚本，输出 `event_timeseries_result.json`（114个事件）+ `news_output.json`（1228篇文章）
4. **4号算法** → 读取 3 号结果 + 1 号传播数据，运行 4 个算法模块，输出 API JSON
5. **5号前端** → 调用 4 号 FastAPI 接口，ECharts/D3 可视化渲染

**功能说明**：实现对 11 个主流信息平台的舆情数据采集，支持关键词搜索、分页爬取、增量采集与历史数据合并，输出统一格式的 JSON/CSV 数据供下游处理。

**覆盖平台**：

| 序号 | 爬虫文件 | 平台 | 技术特点 |
|:---:|------|------|------|

| 3 | `bilibili_crawler.py` | B站 | 视频平台搜索，含转发关系 |
| 4 | `zhihu_crawler.py` | 知乎 | 问答平台搜索 |
| 5 | `toutiao_crawler.py` | 今日头条 | 资讯聚合平台 |
| 6 | `xiaohongshu_crawler.py` | 小红书 | 笔记分享平台 |
| 7 | `newsapi_crawler.py` | NewsAPI | 国际新闻 API |
| 8 | `mediacloud_crawler.py` | MediaCloud | 学术级新闻数据库 |

**核心子模块**：

| 文件 | 功能 |
|------|------|
| `main.py` | 统一命令行入口，支持 `--keywords` `--platforms` `--pages` 参数 |
| `config.py` | 全局配置：数据目录、默认关键词、起始 ID |
| `save_data.py` | 数据持久化：assign_ids（统一编号）、deduplicate（去重）、save_json/save_csv/save_jsonl |
| `search_fallback.py` | 搜索回退策略：主搜索引擎失效时自动切换备用方案 |
| `http_client.py` | HTTP 请求封装：User-Agent 轮换、超时重试、反爬应对 |
| `find_mediacloud_ids.py` | MediaCloud 数据源 ID 探索工具 |

**核心算法流程**：

```
输入: 命令行参数 (keywords, platforms, pages)
  │
  ├─→ Step 1: 解析参数 → 确定采集平台列表和关键词列表
  │
  ├─→ Step 2: 遍历关键词 × 平台：
  │         调用对应爬虫模块
  │         每个平台独立采集，互不阻塞
  │         异常隔离：单平台失败不影响其他平台
  │
  ├─→ Step 3: 本次结果去重 (deduplicate)
  │         基于文本相似度（difflib.SequenceMatcher）
  │
  ├─→ Step 4: 统一编号 (assign_ids)
  │         按全局递增 ID 分配
  │
  ├─→ Step 5: 历史数据合并（默认行为）
  │         读取 fixed_json_path 的已有数据
  │         新旧合并再全局去重
  │         --reset 参数可跳过
  │
  ├─→ Step 6: 多格式输出
  │         批次文件: raw_data_{timestamp}.json/.jsonl/.csv
  │         固定文件: raw_data.json/.csv (后端固定读取)
  │
  输出: raw_data.json, raw_data.csv
```

#### 模块二：数据预处理流水线（2号工程师）

**代码位置**：`scripts/` + `utils/`（15 个 Python 文件）  
**GitHub 分支**：[`2-preprocess`](https://github.com/sherlock211588/opinion_system/tree/2-preprocess)

**功能说明**：7 阶段数据处理流水线，将原始 HTML 网页转换为结构化、可分析的清洗文本数据。

**流水线各阶段**：

| 阶段 | 脚本 | 功能 | 核心技术 |
|:---:|------|------|------|
| 1 | `01_load_data.py` | 数据加载 | 读取 `input/raw_data.json`，支持 JSON/CSV/Excel |
| 2 | `02_extract_body.py` | 正文抽取 | readability-lxml 提取文章正文，剥离广告/导航/推荐 |
| 3 | `03_clean_deduplicate.py` | 清洗与去重 | 18 项清洗操作：BBone 链接清洗、MD5 精确去重、SimHash 近重复检测、Emoji 清洗、OpenCC 繁简转换、广告/声明剥离 |
| 4 | `04_segment.py` | 中文分词 | jieba 分词 + 自定义停用词表过滤 + 自定义用户词典 |
| 5 | `05_feature_extract.py` | 特征提取 | scikit-learn TF-IDF 特征矩阵 |
| 6 | `06_generate_report.py` | 质量报告 | HTML/JSON/TXT 格式数据质量报告 |
| 7 | `07_advanced_analysis.py` | 高级分析 | 情感评分(0-1连续值)、TextRank 摘要、命名实体识别(人名/地名/机构名) |

**工具模块 (`utils/`)**：

| 文件 | 功能 |
|------|------|
| `cleaners.py` | 18 项文本清洗操作集合 |
| `deduplicators.py` | MD5 精确去重 + SimHash 近重复检测 |
| `tokenizers.py` | jieba 分词封装，加载自定义词典和停用词表 |
| `reporters.py` | 多格式数据质量报告生成 |
| `validators.py` | 数据有效性校验 |
| `progress.py` | 进度条与任务追踪 |
| `logger.py` | 统一日志记录 |

**停用词资源**：
- `hit_stopwords.txt` — 哈工大停用词表
- `extra_stopwords.txt` — 舆情领域扩展停用词
- `user_dict.txt` — 自定义用户词典（网络新词、舆情专用词）

**核心算法流程**：

```
输入: raw_data.json (爬虫原始数据)
  │
  ├─→ Stage 1 [load]: 读取原始文件，校验必填字段
  │         (html_content, title, publish_time, source)
  │
  ├─→ Stage 2 [extract]: readability-lxml 正文抽取
  │         识别正文区域 → 剔除广告/导航/推荐/评论区
  │         → 输出纯文本 title + body
  │
  ├─→ Stage 3 [clean]:
  │         ├─ BBone清洗: 去除特殊格式标记
  │         ├─ HTML标签: 彻底清理残留标签
  │         ├─ Emoji清洗: 表情符号转文字或删除
  │         ├─ 繁简转换: OpenCC s2t / t2s
  │         ├─ 广告声明: 正则匹配("免责声明"等) → 剔除
  │         ├─ MD5去重: 全文 MD5 哈希 → 删重复文章
  │         └─ SimHash: 64位海明距离 ≤ 3 → 删近重复文章
  │
  ├─→ Stage 4 [segment]:
  │         jieba.cut() 分词 → 停用词过滤 → 用户词典修正
  │         → 输出空格分隔的分词文本
  │
  ├─→ Stage 5 [feature]: TfidfVectorizer
  │         max_features=5000, ngram_range=(1,2)
  │         → tfidf_matrix.npz
  │
  ├─→ Stage 6 [report]: 数据质量统计
  │         总条数 / 空值率 / 平均长度 / 去重率 / 分词覆盖率
  │
  ├─→ Stage 7 [advanced]:
  │         情感评分 (0~1连续值)
  │         TextRank 自动摘要
  │         NER 命名实体识别 (人名/地名/机构名)
  │
  输出: segmented_data.csv, tfidf_matrix.npz, advanced_analysis.csv
```

**4.1 子模块一：舆情生命周期预测 (`lifecycle/`)**

| 文件 | 核心类/函数 | 职责 |
|------|-----------|------|
| `detector.py` | `LifecycleDetector` | 主调度器，串联全部子模块 |
| `stage.py` | `classify_stage()` | EMA 平滑 + softmax → 四阶段概率分布 |
| `fusion.py` | `compute_heat_index()` | 四维加权融合 → 综合热度指数(0~100) |
| `fusion.py` | `remove_diurnal_cycle()` | 日周期剥离，避免凌晨低谷误判 |
| `forecast.py` | `forecast_holt()` | 自适应 Holt 双指数平滑 → 未来预测 + 95% 置信区间 |
| `changepoint.py` | `find_turning_points()` | PELT 统计变点检测 → 阶段切换拐点 |
| `resurgence.py` | `detect_resurgence()` | 峰-谷-峰三条件联合 → 二次爆发检测 |
| `critical.py` | `detect_critical_slowing_down()` | 方差比 + AR(1) → 临界减速预警 |

**核心算法流程**：

```
输入: event_data {event_id, timeseries[]}
  │
  ├─→ Step 1: _trim_padding()  裁剪头尾零值填充
  │
  ├─→ Step 2: _assess_data_quality()  数据质量四级分级
  │         full(≥12点) / basic(8-11点) / minimal(4-7点) / sparse(<4点)
  │         后续步骤按等级自适应启用/降级
  │
  ├─→ Step 3: remove_diurnal_cycle()  日周期剥离
  │         按间隔分桶统计每桶平均 → 计算偏离因子
  │         振幅 ≥ 0.30 → 逐条除以因子拉平 (basic/full 级)
  │
  ├─→ Step 4: compute_heat_index()  四维融合
  │         报道量(0.40) + 情感分歧度(0.25) + 平台扩散度(0.20) + 互动热度(0.15)
  │         → 综合热度指数 composite[]
  │
  ├─→ Step 5: _calc_trend()  趋势判定
  │         前窗口非零均值 vs 后窗口非零均值 → 斜率
  │         slope > +0.15 → "上升" / slope < -0.10 → "下降" / else → "平稳"
  │
  ├─→ Step 6: classify_stage()  阶段分类
  │         三信号 → raw scores → EMA平滑(α=0.30) → softmax(T=0.50) → 四阶段概率
  │         raw_latent  = (1-level)×(1-|trend|)×(1-time)
  │         raw_growing = max(0,trend)×level×(1-time)
  │         raw_peak    = level×(1-|trend|)×time
  │         raw_decline = max(0,-trend)×(0.3+0.7×time)
  │         → 取最大概率的阶段标签
  │
  ├─→ Step 7: detect_critical_slowing_down()  临界减速预警
  │         variance_ratio = 最近窗方差/基线方差
  │         ar1 = 自回归系数
  │         → none(yellow/orange/red (basic/full 级)
  │
  ├─→ Step 8: detect_resurgence()  二次爆发 (basic/full 级)
  │         三条件: 第一峰在前65% + 反弹≥3.0x + 绝对增量≥45
  │
  ├─→ Step 9: find_turning_points()  PELT变点检测 (basic/full 级)
  │         pen = 1.5×log(n)×σ², n<10 回退差分法
  │
  ├─→ Step 10: forecast_holt()  自适应Holt预测 (full 级)
  │         6×5 网格自动搜最优(α,β) → 双路并行 Holt
  │         短期(12h窗口)+长期(60%窗口) → MSE反比加权融合
  │         → 预测值 + 1.96σ 置信区间
  │
  输出: {current_stage, stage_probabilities, predicted_next_24h,
         turning_points, critical_early_warning, resurgence, ...}
```

**4.2 子模块二：传播溯源 (`propagation/tracer.py`)**

| 核心函数 | 功能 | 算法描述 |
|---------|------|---------|
| `analyze()` | 主入口 | 建图→找源头→图算法→深度→触达→时间线→反事实→可视化 |
| `_build_graph()` | 构建有向传播图 | nx.DiGraph, parent→child 加边, 标记孤立节点, DAG 检测 |
| `_find_sources()` | 找源头节点 | 入度=0 的节点, 按时间排序 |
| `_find_key_nodes_graph()` | 三维图算法关键节点 | PageRank(0.45) + 介数中心性(0.35) + 出度(0.20) → 综合分, 取前40% |
| `_classify_node_role()` | 自动节点角色标注 | 6 类: 初始爆料源/超级传播节点/跨平台桥接/官媒/大V/参与者 |
| `_counterfactual_analysis()` | 反事实推演 | G.remove_node(nid) → 重算 BFS → 损失 = baseline - cf → impact_ratio |
| `_calc_total_reach()` | 去重触达 | 平台内指数衰减 fan×(1-0.40)^i + 转发曝光 fwd×50 |
| `_build_graph_for_viz()` | 前端可视化数据 | symbolSize=15+PR×100(15~80px), 5 类颜色分类 |

**核心算法流程**：

```
输入: nodes[{node_id, parent_node_id, follower_count, is_verified, ...}]
  │
  ├─→ Step 1: _build_graph()
  │         nx.DiGraph(), add_node()×N, add_edge()×M
  │         标记孤立节点 (父节点不在集合中)
  │
  ├─→ Step 2: _find_sources()  入度=0 → 源头列表
  │
  ├─→ Step 3: _find_key_nodes_graph()
  │         PageRank + 介数中心性 + 出度 归一化
  │         综合 = 0.45PR + 0.35BC + 0.20OD
  │         → 取前 40% → 角色自动标注
  │
  ├─→ Step 4: _calc_graph_depth()  最长传播链 (DAG拓扑/非DAG BFS)
  │
  ├─→ Step 5: _calc_total_reach()  去重触达估算
  │
  ├─→ Step 6: _counterfactual_analysis()  ★ 核心创新
  │         baseline_BFS → remove_node → cf_BFS
  │         impact_ratio = (baseline_reach - cf_reach) / baseline_reach
  │
  ├─→ Step 7: _build_graph_for_viz()  力导向图数据
  │
  输出: {source_nodes, key_nodes, propagation_depth, total_reach,
         counterfactual_analysis, graph_for_visualization, ...}
```

```

  │

  │     联合模型: TF-IDF(2000维) + 元数据(8维) → LogisticRegression

【推理阶段】
  │

  │     有元数据 → 联合模型 / 无元数据 → 纯文本模型
  │

  │     10维拓扑特征 → reinforce_real / reinforce_fake / neutral
  │     → 四个修正逻辑调整最终判定
  │
```

**4.4 子模块四：跨事件因果分析 (`cross_event.py`)**

| 核心函数 | 功能 | 算法描述 |
|---------|------|---------|
| `analyze()` | 主入口 | 遍历事件对 → 格兰杰检验 + 传递熵 → 构建因果图 |
| `_granger_test()` | 格兰杰因果检验 | OLS 手写回归 → Restricted vs Unrestricted → F 统计量 → p 值 |
| `_ols()` | 最小二乘回归 | β = (XᵀX)⁻¹Xᵀy，奇异时伪逆回退 |
| `_f_pvalue()` | F 分布 p 值 | scipy.f.sf；回退：不完全 Beta 函数连分式展开 |
| `_symbolic_transfer_entropy()` | 符号化传递熵 | 分位数离散化 → 转移概率矩阵 → TE 公式 → 200 次 shuffle 检验 |

**核心算法流程**：

```
输入: events {event_id: [timeseries]}
  │
  ├─→ 提取时序: driver = negative_ratio, target = news_count
  │
  ├─→ 格兰杰因果 (逐对事件 A→B):
  │     对 lag ∈ [1, max_lag]:
  │       Restricted:   Y[t] = α + Σβ_i×Y[t-i]
  │       Unrestricted: Y[t] = α + Σβ_i×Y[t-i] + Σγ_i×X[t-i]
  │       OLS → RSS_r, RSS_u
  │       F = ((RSS_r-RSS_u)/L) / (RSS_u/(n-2L-1))
  │       → p 值 (F 分布右尾)
  │     → 取最小 p 值的最优 lag
  │     → p < 0.05 → 显著线性因果
  │
  ├─→ 符号化传递熵 (逐对事件 A→B):
  │     1. 按 33%/67% 分位数 → 符号化(低/中/高)
  │     2. 计算转移概率 p(y_{t+1}|y_t,x_t) vs p(y_{t+1}|y_t)
  │     3. TE = Σ p_j × log(p_cond_joint / p_cond_marg)
  │     4. 200 次随机打乱 → 零分布 → p 值
  │     TE_effective = max(0, TE_raw - mean(TE_shuffled))
  │     → p < 0.05 → 显著非线性因果
  │
  输出: {pairs(格兰杰), transfer_entropy_pairs(TE), causal_graph}
```

## 四、系统设计难点与亮点

### 4.1 难点

**难点1：多平台爬虫的反爬对抗与数据格式统一**
11 个平台各自有不同的页面结构、API 接口和反爬策略。解决方案：每个平台独立的爬虫模块，异常隔离（单平台失败不影响其他平台）；`http_client.py` 封装 User-Agent 轮换和超时重试；`search_fallback.py` 实现搜索引擎失效时的自动切换。

**难点2：网页正文抽取的准确性**
不同网站的 HTML 结构差异巨大，正文区域、广告、导航、评论区混杂。解决方案：采用 readability-lxml 的启发式算法识别正文区域，配合自定义的正则规则剥离特定网站的广告模板和声明文字。

**难点3：虚假检测中"信息不足"与"信息虚假"的区分**
传统二分类（真/假）将"缺乏证据"的突发事件误判为假。解决方案：设计三元分类器（可信/待验证/疑似虚假），引入信息充分度评估和不确定语言保护机制，区分"没证据证明真"和"有证据证明假"。同时设计双模型路由——元数据充足时用联合模型(2008维)，不足时自动切换纯文本模型(2000维)，避免默认值系统性偏判。

**难点4：传播图中"小粉丝大影响"节点的识别**
传统方法按粉丝数排序关键节点，会忽略粉丝少但结构关键的桥接节点。解决方案：引入 PageRank(全局影响力) + 介数中心性(桥接作用) + 出度(引爆能力) 三维图算法综合评分，并通过反事实推演量化每个节点的因果影响力——模拟移除节点后传播规模损失多少。

**难点5：跨事件因果推断的可靠性**
格兰杰因果检验假设线性关系，无法捕捉非线性因果。解决方案：同时运行格兰杰因果（线性）和符号化传递熵（非线性，时序离散化后计算信息转移概率，200次shuffle显著性检验），双算法交叉验证，提高因果推断的可信度。

### 4.2 亮点

**亮点1：临界减速预警（CSD）**
将复杂系统理论中的"相变临界减速"现象迁移到舆情分析——系统从潜伏期跳转到成长期之前，方差会先增大、自相关会增强。根据方差比和 AR(1)系数四级预警，可在热度尚未明显上涨时提前 4~6 小时预警爆发风险。

**亮点2：反事实推演量化因果影响力**
不只看"谁的粉丝多"，而是模拟"如果这个节点不存在，传播规模会损失多少"。这是来自 EPFL 因果推断研究的工程实现，比传统的 PageRank/粉丝数更能回答"谁真正推动了传播"。

**亮点3：传播拓扑特征辅助虚假判别**
真实新闻和假新闻的传播图拓扑存在系统性差异（多源头浅扩散 vs 单源头深传播）。从传播图中提取 10 维拓扑特征（源头数/深度/密度/聚类系数/出度分布/平台数/跨平台边比/粉丝基尼系数），对文本模型判定进行二次修正。基于 KDD 2024(PSGT) 和 AAAI 2023(Zero-Shot) 顶会论文。

**亮点4：SHAP 词级可解释性**
不是黑盒输出"虚假概率"，而是用 SHAP 值逐词解释判定依据——哪些词推高了虚假概率（红色高亮）、哪些词提升了可信度（绿色高亮）。基于 I-FLASH (Dua et al., 2023)的解析解 SHAP，毫秒级计算，不依赖采样。

**亮点5：双算法交叉验证的跨事件因果**
格兰杰因果（线性） + 符号化传递熵（非线性）双管齐下。传递熵不需要假设线性关系，弥补了格兰杰因果的盲区。两套算法结果互补，使因果推断更可靠。

**亮点6：四级数据质量自适应降级**
不是粗暴地对所有事件跑全量算法，而是根据数据充足度分级启用功能：full(全功能:预测+预警) → basic(阶段+趋势+变点) → minimal(基础研判) → sparse(仅阶段标签)。每个接口返回都包含数据质量等级，前端可据此调整展示内容。

## 六、设计心得

1. **分层架构是大型系统的基石**：从数据采集→预处理→NLP→算法→API→前端的六层架构，使 5 人团队能够完全并行开发。统一 JSON Schema 约定是各层之间唯一的耦合点——任何人只要遵守输入输出格式，就可以独立开发和测试。

2. **"数据不足时不硬算"比"什么都能算"更重要**：在 114 个真实事件中，质量参差不齐。与其对所有事件跑全量算法输出不可靠的结果，不如诚实分级：full 级给全功能，sparse 级只给阶段标签。这比追求 100% 覆盖率更工程化。

3. **规则兜底是 AI 系统的必要组件**：智能报告生成模块设计了 LLM 和规则两种模式——即使没有 API Key 或网络不通，系统仍能用模板生成可用报告。这种"先有兜底，再追求更好"的思路值得推广。

4. **算法的可解释性是实际部署的前提**：如果虚假检测只输出一个"假"字而没有解释，用户不会信任。SHAP 词级解释、反事实推演的文字描述、临界预警的具体指标——每个输出都要回答"为什么"。

5. **团队协作中，接口文档先行**：4 号在代码完成前就写好了 JSON 格式约定文档。这使得 5 号可以先用 Mock 数据开发前端界面，后端完成后直接对接，效率远高于"写完代码再补文档"。

6. **模块独立自测**：每个 `.py` 文件末尾都有 `if __name__ == "__main__"` 的自测代码，每个模块都可以独立运行和调试，不需要启动整个系统即可验证单个算法的正确性。这极大地提升了开发效率。
