## 一、课程设计目的

本课程设计旨在综合运用 Python 程序设计、数据结构与算法、Web 开发、机器学习、自然语言处理、图算法、时序分析等课程知识，由 5 人团队协作构建一个完整的**网络舆情事件智能分析系统**。

系统的核心业务流程为：从微博、B站、知乎、今日头条等 11 个信息源自动采集舆情数据，经 HTML 正文抽取、中文清洗去重、jieba 分词、TF-IDF 特征提取后，利用 BGE 深度学习模型进行文本向量化、HDBSCAN 密度聚类实现事件聚合、中文情感三分类模型完成情感推理、大语言模型进行事件要素结构化抽取，最终由时序算法层完成舆情生命周期预测、传播路径溯源、虚假文本检测和跨事件因果分析，所有分析结果通过 FastAPI Web 服务以 JSON 格式输出，由 Vue 3 + ECharts + D3.js 前端进行可视化渲染。

通过本项目的实践，达到以下目标：

1. **掌握大型软件系统的六层架构设计**：从数据采集层、预处理层、NLP 语义层、算法分析层、Web 服务层到前端展示层的完整技术栈；
2. **实践多种机器学习和统计方法在真实场景中的工程化应用**：包括 HDBSCAN 聚类、BGE 向量嵌入、FAISS 语义检索、TF-IDF 特征工程、逻辑回归分类、Holt 指数平滑预测、PageRank 图算法、格兰杰因果检验与符号化传递熵等；
3. **培养团队协作与工程化思维**：5 人团队按模块并行开发，通过统一 JSON Schema 实现跨模块数据对接，体验真实工业级协作流程；
4. **理解算法可解释性的工程实现**：SHAP 词级解释、反事实推演因果量化、临界减速预警等，使黑盒算法的输出对用户可理解、可追溯。

## 二、需求分析

### 2.1 项目背景

当前突发公共事件、社会热点话题在社交平台上传播迅速，舆情信息可能在短时间内对社会稳定产生重大影响。如何从海量多源信息中自动识别热点事件、分析传播趋势、发现虚假信息、预判舆情走向，是网络空间治理的现实需求。

### 2.2 功能需求

依据课程要求，系统需实现 **7 项基础功能 + 2 项高级功能 + 创新设计**：

| 序号 | 功能模块 | 需求级别 | 说明 |
|:---:|------|:---:|------|
| 1 | 多平台网络爬虫 | 基础(1) | 从主流新闻网站、社交平台采集舆情数据 |
| 2 | 数据清洗与预处理 | 基础(2) | 去重、去噪、格式标准化 |
| 3 | 内容分析 | 基础(3) | 网页正文抽取、中文分词、TF-IDF 特征表示 |
| 4 | 热点事件发现 | 基础(4) | 利用机器学习方法识别热点事件 |
| 5 | 舆情事件聚合 | 基础(5) | 同一事件多条报道聚合，支持相似事件检索 |
| 6 | 情感倾向分析 | 基础(6) | 正面/负面/中性情感计算 |
| 7 | 舆情生命周期预测 | 基础(7) | 预测事件所处阶段，趋势预测 |
| 8 | 虚假文本检测 | 高级(1) | 对采集信息进行真伪判断，给出置信度 |
| 9 | 传播路径溯源 | 高级(2) | 识别关键传播节点，构建传播路径图 |
| 10 | 跨事件因果分析 | 创新 | 发现多事件间因果传导关系 |

### 2.3 界面需求

| 界面 | 功能描述 |
|------|------|
| 系统首页 / 登录 | 用户名密码登录 |
| 舆情事件看板 | 热点事件列表（时间线/卡片），含热度指数、情感倾向、预警灯，支持排序筛选 |
| 事件详情分析报告 | 事件概述、发展趋势图、情感分布图、平台分布、高频关键词、传播路径图 |
| 智能问答 | 接入大模型对当前事件进行自由提问 |
| 个人中心 | 配置关注平台、管理关键词 |

### 2.4 非功能需求

- **性能**：算法模块纯 CPU 运行，不依赖 GPU；模型文件 < 100KB（虚假检测），向量模型约 1.3GB；首次启动含模型加载约 30 秒；
- **可靠性**：数据质量四级自动分级，低质量数据自动降级算法功能，不崩溃；
- **可扩展性**：各层之间通过 JSON Schema 松耦合，每层独立可替换；
- **可解释性**：所有算法判定附带可追溯的解释依据。

### 2.5 开发环境

| 类别 | 技术选型 |
|------|------|
| 爬虫 | Python requests, BeautifulSoup, JSON/CSV |
| 预处理 | Python pandas, readability-lxml, jieba, OpenCC, scikit-learn, SnowNLP, TextRank4ZH |

| 聚类 | HDBSCAN |
| 向量检索 | FAISS (IVF 倒排索引) |
| 时序算法 | numpy, scipy, scikit-learn, networkx, ruptures |
| Web 服务 | FastAPI + Uvicorn |
| 前端 | Vue 3 + ECharts + D3.js |
| 版本管理 | Git + GitHub（三分支并行开发） |

#### 模块一：多平台网络爬虫 —— 1号工程师

**代码位置**：`crawler/`（21 个文件，约 1500 行 Python）  
**负责功能**：后端基础功能(1) — 网络爬虫

**功能说明**：实现对 11 个主流信息平台的舆情数据采集，通过统一命令行入口 `main.py` 驱动，支持关键词搜索、分页爬取、增量采集与历史数据合并，输出标准化 JSON/CSV 数据供 2 号预处理模块消费。

**覆盖的 11 个平台**：

| 序号 | 爬虫文件 | 目标平台 | 数据特点 |
|:---:|------|------|------|
| 1 | `news_crawler.py` | 百度新闻搜索 | 通用新闻聚合，支持详情页抓取 |
| 2 | `weibo_crawler.py` | 微博 | 社交媒体短文本，高互动 |
| 3 | `bilibili_crawler.py` | B站 | 视频平台，含转发关系链（1号→4号传播溯源关键数据） |
| 4 | `zhihu_crawler.py` | 知乎 | 长文问答平台 |
| 5 | `toutiao_crawler.py` | 今日头条 | 推荐算法驱动的资讯平台 |
| 6 | `xiaohongshu_crawler.py` | 小红书 | 笔记分享平台 |
| 7 | `newsapi_crawler.py` | NewsAPI | 国际新闻 API 接口 |
| 8 | `mediacloud_crawler.py` | MediaCloud | 学术级全球新闻数据库 |
| 9 | `currents_crawler.py` | Currents API | 多语言新闻聚合 |
| 10 | `rss_news_crawler.py` | RSS 多源聚合 | XML 解析 + 去重 |
| 11 | `site_news_crawler.py` | 指定媒体站点 | 定向深度采集 |

**核心子模块清单**：

| 文件 | 核心功能 | 关键实现 |
|------|------|------|
| `main.py` | 统一命令行入口，调度全部爬虫 | argparse 参数解析(`--keywords`/`--platforms`/`--pages`/`--reset`)，遍历关键词×平台调用爬虫，异常隔离 |
| `config.py` | 全局配置中心 | 数据目录、默认关键词列表、起始 ID、默认页数 |
| `save_data.py` | 数据持久化与质量控制 | `assign_ids()` 全局递增编号、`deduplicate()` 基于 difflib.SequenceMatcher 文本去重、`save_json/save_csv/save_jsonl` 多格式输出 |
| `search_fallback.py` | 搜索回退策略 | 主搜索引擎失效时自动切换备用方案 |
| `http_client.py` | HTTP 请求封装 | User-Agent 轮换、超时重试、异常处理 |
| `find_mediacloud_ids.py` | MediaCloud 数据源探索 | 约 500 行，自动发现和验证数据源 ID |

**核心算法流程**：

```
输入: 命令行参数 (keywords[], platforms[], pages, --reset)
  │
  ├─→ Step 1: parse_args() 解析参数
  │         确定爬取平台白名单、关键词列表、翻页深度
  │
  ├─→ Step 2: 双重循环 (keyword × platform)
  │         ├─ 动态导入对应爬虫模块
  │         ├─ 调用 crawl(keyword, max_pages=pages)
  │         ├─ 异常隔离：try/except，单平台失败不阻断其余
  │         └─ 日志记录：每个平台返回条数
  │
  ├─→ Step 3: 本次结果处理
  │         ├─ deduplicate() — 基于文本相似度的内部去重
  │         └─ assign_ids() — 从 start_id 起全局递增编号
  │
  ├─→ Step 4: 历史数据合并（默认，--reset 跳过）
  │         ├─ load_existing_records(fixed_json_path)
  │         ├─ old + new → 合并去重
  │         └─ 重新 assign_ids
  │
  ├─→ Step 5: 多格式持久化
  │         批次文件: raw_data_{timestamp}.json/.jsonl/.csv
  │         固定文件: raw_data.json/.csv (2号固定读取)
  │
  输出: raw_data.json, raw_data.csv
```

#### 模块二：数据预处理流水线 —— 2号工程师

**代码位置**：`scripts/` + `utils/`（15 个 Python 文件，约 2000 行）  
**GitHub 分支**：[`2-preprocess`](https://github.com/sherlock211588/opinion_system/tree/2-preprocess)  
**负责功能**：后端基础功能(2)(3) — 数据清洗与预处理、内容分析

**功能说明**：实现一个 7 阶段数据处理流水线，将 1 号爬虫输出的原始 HTML 网页数据转换为结构化、可直接供 NLP 模型消费的清洗文本和特征矩阵。流水线设计为**可独立运行的单步脚本**，通过 `run_all.py` 串联，支持 `--from` 和 `--skip` 参数灵活控制执行范围。

**流水线各阶段**：

| 阶段 | 脚本 | 功能 | 核心技术 |
|:---:|------|------|------|
| 1 | `01_load_data.py` | 原始数据加载与校验 | 读取 `input/raw_data.json`，校验必填字段(`html_content`,`title`,`publish_time`,`source`) |
| 2 | `02_extract_body.py` | 网页正文抽取 | **readability-lxml** 启发式算法识别正文区域，自动剥离广告、导航栏、评论区、推荐列表 |
| 3 | `03_clean_deduplicate.py` | 综合清洗与去重 | **18项清洗操作**：BBone格式清洗、残余HTML标签清理、Emoji表情处理、**OpenCC繁简转换**、广告声明正则剥离、非中文内容过滤、短文本剔除、**MD5精确去重** + **SimHash近重复检测**(海明距离≤3) |
| 4 | `04_segment.py` | 中文分词 | **jieba** 分词 + 哈工大停用词表过滤 + 自定义用户词典(舆情领域新词) |
| 5 | `05_feature_extract.py` | TF-IDF 特征提取 | scikit-learn `TfidfVectorizer(max_features=5000, ngram_range=(1,2))` |
| 6 | `06_generate_report.py` | 数据质量报告 | HTML/JSON/TXT 多格式，含总量/空值率/平均长度/去重率/分词覆盖率 |
| 7 | `07_advanced_analysis.py` | 高级文本分析 | **SnowNLP** 情感评分(0~1连续值 + 三分类标签)、**TextRank4ZH** 自动摘要(取Top-3句)、**jieba.posseg** 词性标注实现命名实体识别(人名nr/地名ns/机构名nt+nz, Top-5) |

**工具模块 (`utils/`)**：

| 文件 | 职责 | 关键函数 |
|------|------|------|
| `cleaners.py` | 18项文本清洗操作集合 | `clean_all(text)` 链式调用全部清洗规则 |
| `deduplicators.py` | 文本去重 | `deduplicate(df, col)` — MD5精确匹配 + SimHash 64位近重复检测 |
| `tokenizers.py` | 中文分词封装 | `EnhancedTokenizer` 类，加载自定义词典和停用词表，`segment(text, pos_filter)` |
| `reporters.py` | 质量报告生成 | `generate_quality_report(original, cleaned, stats, path)` |
| `validators.py` | 数据校验 | `validate_dataframe(df, required_cols)` / `assert_data_quality(df, col)` |
| `progress.py` | 进度可视化 | `process_with_progress(items, func)` — tqdm 进度条封装 |
| `logger.py` | 统一日志 | `setup_logger(name, log_file)` 控制台+文件双输出 |

**输入输出契约**：

| 方向 | 文件 | 格式 | 关键字段 |
|------|------|------|------|
| 输入(来自1号) | `input/raw_data.json` | JSON数组 | `html_content`, `title`, `publish_time`, `source` |
| 输出(给3号) | `output/cleaned_data.csv` | CSV | `news_id`, `title`, `content`(清洗正文), `segmented`(分词), `publish_time`, `source` |
| 输出(给3/4/5号) | `output/advanced_analysis.csv` | CSV | 以上全部 + `sentiment_polarity`, `sentiment_label`, `summary`, `persons`, `locations`, `orgs` |
| 输出(给3号) | `output/tfidf_matrix.npz` | NPZ | 稀疏TF-IDF特征矩阵 |

**核心算法流程**：

```
输入: raw_data.json (1号爬虫原始数据)
  │
  ├─→ Stage 1 [load]: 文件格式校验 → 必填字段检查 → 空值过滤
  │
  ├─→ Stage 2 [extract]: readability-lxml 正文抽取
  │         HTML → DOM树解析 → 启发式正文识别 → 纯文本 body
  │         剥离: 导航栏/侧边栏/评论区/广告/推荐链接
  │
  ├─→ Stage 3 [clean]: 18项清洗管道
  │         ├─ BBone格式清洗 → 残余HTML标签清理
  │         ├─ Emoji处理 → OpenCC繁简转换(s2t/t2s)
  │         ├─ 广告声明正则匹配("免责声明"/"广告"等) → 剔除
  │         ├─ 非中文内容过滤(中文字符占比检测)
  │         ├─ 短文本过滤(len < MIN_TEXT_LENGTH)
  │         ├─ MD5精确去重(全文哈希碰撞 → 删除重复)
  │         └─ SimHash近重复检测(64位海明距离 ≤ 3 → 合并)
  │
  ├─→ Stage 4 [segment]: jieba 中文分词
  │         cut(text) → 停用词表过滤(哈工大+自定义)
  │         → 用户词典修正(舆情专用词/网络新词)
  │         → 空格分隔的分词文本 + 词数统计
  │
  ├─→ Stage 5 [feature]: TF-IDF 特征矩阵
  │         TfidfVectorizer(max_features=5000, ngram_range=(1,2))
  │         → 稀疏矩阵 → tfidf_matrix.npz + vectorizer.joblib
  │
  ├─→ Stage 6 [report]: 数据质量报告生成
  │
  ├─→ Stage 7 [advanced]: 三项高级分析
  │         ├─ SnowNLP 情感评分: polarity(0~1) → 正面(>0.65)/负面(<0.35)/中性
  │         ├─ TextRank4ZH 摘要: 构建词图 → PageRank → 取Top-3关键句
  │         └─ jieba.posseg NER: nr→人名 / ns→地名 / nt+nz→机构名, Top-5去重
  │
```

#### 4.4 子模块：跨事件因果分析

**文件**：`cross_event.py`（621行，纯手写实现，零外部统计库依赖）

| 核心函数 | 功能 | 算法描述 |

| `_granger_test()` | 格兰杰因果 | OLS(Restricted vs Unrestricted) → F统计量 → p值, 各lag取最优 |
| `_ols()` | 最小二乘 | β=(XᵀX)⁻¹Xᵀy, 奇异→伪逆回退 |
| `_f_pvalue()` | F分布p值 | scipy.f.sf; 回退: 不完全Beta函数连分式展开 |
| `_symbolic_transfer_entropy()` | 符号化传递熵 | 33%/67%分位数离散化→转移概率矩阵→TE公式→200次shuffle检验→TE_effective |

**核心算法流程**：

```
输入: events {event_id: [timeseries]}
  │
  ├─→ 格兰杰因果 (逐对A→B, lag∈[1,max_lag]):
  │     Restricted: Y[t] = α+ΣβY[t-i]    vs   Unrestricted: +ΣγX[t-i]
  │     OLS → RSS_r, RSS_u → F = ((RSS_r-RSS_u)/L)/(RSS_u/(n-2L-1))
  │     → p值(F分布右尾) → 取最小p的最优lag → p<0.05显著
  │
  ├─→ 符号化传递熵 (逐对A→B):
  │     时序→分位数离散化(低/中/高) → 联合转移概率
  │     TE = Σp_j×log(p_cond_joint/p_cond_marg)
  │     200次shuffle→零分布→p值 → TE_effective = max(0,TE_raw-mean(TE_shuff))
  │
  输出: {pairs(格兰杰), transfer_entropy_pairs(TE), causal_graph,
         significant_pairs: 2510/4692}
```

## 四、系统设计难点与亮点

### 4.1 难点

**难点1：多平台爬虫的反爬对抗与数据格式统一（1号）**
11 个平台各自有不同的页面结构、API 接口和反爬策略。解决方案：每个平台独立的爬虫模块，异常隔离（单平台失败不影响其余）；`http_client.py` 封装 User-Agent 轮换和超时重试；`search_fallback.py` 实现搜索引擎失效时的自动切换。

**难点2：网页正文抽取的准确性（2号）**
不同网站的 HTML 结构差异巨大，正文区域、广告、导航、评论区混杂。解决方案：readability-lxml 启发式算法识别正文区域 + 自定义正则规则剥离特定网站广告模板。输出前通过 `assert_data_quality()` 验证正文长度。

**难点3：HDBSCAN 事件聚合的粒度控制（3号）**
聚类太松→不同事件被错误合并；聚类太紧→同一事件被拆分。解决方案：通过 `min_cluster_size=5` 和 `min_samples=3` 的调参，结合 `cluster_selection_method="eom"` 自动确定最优簇数，噪声点标记为 `EVT_NOISE` 进入候选池。

**难点4：6小时间隔时序数据的零值填充（3号）**
舆情事件存在凌晨报道空白期和事件间歇期，直接聚合会产生不连续时间轴。解决方案：`fill_empty_slice_data()` 从事件首发时间到当前时刻，逐6小时槽填充零值，保证所有事件共享统一连续的时间轴。

**难点5：虚假检测中"信息不足"与"信息虚假"的区分（4号）**
传统二分类将"缺乏证据"误判为假。解决方案：三元分类器（可信/待验证/疑似虚假）+ 双模型路由（联合2008维 / 纯文本2000维）+ 后置降级（短文本+元数据不足→待验证）+ 不确定语言保护（"听说/可能/求证"等）。

**难点6：传播图中"小粉丝大影响"节点的识别（4号）**
传统方法按粉丝数排序，忽略结构关键节点。解决方案：PageRank(0.45) + 介数中心性(0.35) + 出度(0.20) 三维综合评分 + 反事实推演量化因果影响力。

**难点7：跨事件因果推断的可靠性（4号）**
格兰杰因果假设线性关系。解决方案：同时运行格兰杰（线性）和符号化传递熵（非线性，时序离散化+200次shuffle），双算法交叉验证。

### 4.2 亮点

**亮点1：六层松耦合架构**
数据采集→预处理→NLP→算法→API→前端的六层设计使5人完全并行开发。JSON Schema 是唯一耦合点，各层独立可替换。

**亮点2：HDBSCAN 自适应聚类（3号）**
相比传统 K-Means（需预设簇数），HDBSCAN 自动确定最优簇数，且显式处理噪声点。`eom`(excess of mass)方法在舆情场景下比 `leaf` 方法更稳定。

**亮点3：四因子热点评分（3号）**
不只看报道量，综合时间衰减(指数)、媒体多样性(归一化)、情感分歧度(neg权重大)，更真实反映舆情实际烈度。

**亮点4：临界减速预警 CSD（4号）**
将复杂系统相变理论迁移到舆情分析——方差比+AR(1)分级预警，热度未涨时提前4~6小时预判爆发。

**亮点5：反事实推演因果量化（4号）**
模拟"如果这个节点不存在"→计算传播损失→这才是真实因果贡献。EPFL CausalRank 的工程实现。

**亮点6：SHAP 词级可解释性（4号）**
不是黑盒输出概率，而是逐词标注哪些词推高/降低了虚假嫌疑。逻辑回归的解析解SHAP，毫秒级计算。

**亮点7：双算法交叉验证因果（4号）**
格兰杰（线性）+ 符号化传递熵（非线性），双管齐下捕捉不同类型因果。传递熵手写实现，零外部统计库依赖。

**亮点8：四级数据质量自适应降级（4号）**
full(全功能)→basic(阶段+趋势+变点)→minimal(基础研判)→sparse(仅阶段标签)。数据不足时不硬算，诚实降级。

## 六、设计心得

1. **分层架构是大型系统的基石**：从数据采集→预处理→NLP→算法→API→前端的六层架构，使5人团队能够完全并行开发。统一 JSON Schema 约定是各层之间唯一的耦合点——任何人只要遵守输入输出格式，就可以独立开发和测试。实践验证了"高内聚、低耦合"的设计原则。

2. **"数据不足时不硬算"比"什么都能算"更重要**：114个事件质量参差不齐。4号实现了四级数据质量自动评估，数据足的给全功能（63个），数据少的只给基础研判。这比追求100%覆盖率更工程化，也避免了"Garbage In, Garbage Out"。

3. **3号的时序零值填䶿是承上启下的关键设计**：`fill_empty_slice_data()` 虽然只是一个辅助函数，但它保证了114个事件拥有统一连续的时间轴——这是4号所有时序算法能正常运行的前提。这个细节体现了3号对下游需求的深入理解。

4. **规则兜底是 AI 系统的必要组件**：4号的智能报告模块设计了 LLM 和规则两种模式——即使没有 API Key 或网络不通，系统仍能用模板生成可用报告。"先有兜底，再追求更好"的思路值得在AI功能中推广。

5. **算法的可解释性是实际部署的前提**：虚假检测如果只输出一个"假"字而没有解释，用户不会信任。SHAP 词级解释、反事实推演的文字描述、临界预警的具体指标——每个输出都要回答"为什么"。

6. **模块独立自测极大地提升了开发效率**：每个 `.py` 文件末尾都有 `if __name__ == "__main__"` 的自测代码。4号的 `demo.py` 甚至不依赖任何外部数据，用纯模拟数据即可演示全部功能。这使得算法工程师可以在队友数据到位前完成开发和验证。

7. **接口文档先行，对接零摩擦**：4号在代码完成前就写好了 5号前端对接文档，包含完整 JSON Schema + 图表映射。5号用 Mock 数据并行开发前端，后端完成后一行代码对接成功。
